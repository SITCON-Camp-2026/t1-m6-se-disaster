# 需求取捨決策

> 這份文件是根據使用者訪談與 sub-agent 回饋整理出的 v1 取捨草稿，保留人類確認與決策脈絡。

## 我決定 v1 優先服務的使用者

- [x] 行動者

原因：

這個原型的核心目標不是直接派工，而是協助準備投入現場的人先判斷「我現在能不能去」「去了會不會白跑」「還缺哪些確認」。對行動者而言，最重要的是快速看懂需求、需要的人員種類、是否可能已有人處理、目前風險與可信度原因，以及下一步該先確認什麼。

## 這個使用者最想完成什麼

- 快速判斷一筆原始資訊是否足夠支持自己採取下一步。
- 知道目前可能需要哪一類人員或資源。
- 看出是否已有他人回報、討論或可能處理，避免重複前往或做白工。
- 保留哪些資訊仍未確認，而不是將它們誤認為已確認事實。

## 這個使用者最怕什麼錯誤

- 把不確定資訊直接當成可以前往的任務。
- 到現場後才發現需求已改變、物資不收、地點不適合停留，或已有其他人處理。
- 因為看不出風險與缺漏，誤以為某些訊息已被確認。

## 我今天暫時不處理的需求

- 完整的任務派工流程。
- 真實地圖、定位與外部查核資料。
- 讓回報者一次填完所有欄位的完整表單流程。
- 替行動者自動決定是否應該前往現場。

## 我不採用的 AI 建議

- AI 建議：把所有內容直接整理成候選任務清單。
  - 不採用原因：這會把「待確認」與「可行動」混在一起，會放大誤判風險。

## 必須保留人工確認的判斷

- 是否足夠確認地點、時間與需求類型。
- 是否真的缺人或缺物資，以及是否已有人處理。
- 哪些資訊應該保留為原始內容，而不是補成正式結論或可行動任務。
- 什麼時候值得把訊息轉成下一步建議，什麼時候應該停在「待人工確認」。

## 今天明確不做什麼

- 不把未確認內容標成已確認。
- 不在沒有來源證據時自動補完缺失欄位。
- 不把 v1 做成真正的派工系統。

## 我仍不確定的地方

- v1 是否要同時保留「資訊整理者」入口，還是先專注在「行動者判讀」這個核心視角。
- 哪些欄位最值得在介面上以更強的方式提示使用者「這裡還不能直接採取行動」。

# 使用者訪談紀錄

> 這份文件以 release kit 中的 persona 與 sub-agent prompt 為基底，整理出三種使用者觀點的回饋。內容保留人類判斷痕跡，並沒有把未確認資訊直接寫成已確認事實。

## 我的 Phase 0 prototype 現況

目前的原型提供兩個主要視角：原始資訊頁與整理工作台。使用者可以先看到每筆訊息的來源、更新時間與查核狀態，再進一步在整理工作台中看見安全邊界與草稿欄位。這個版本的優勢是能把「不確定」與「待人工確認」顯示出來，但對使用者來說，還不夠清楚哪些內容可以直接行動、哪些只適合留存。

## Persona 1：回報者

### 我想完成的事

- 把我看到或聽到的資訊快速送出去。
- 不想因為表單太嚴格而放棄回報。

### 我看不懂的地方

- 目前畫面沒有很清楚地分出「原始訊息」與「整理後判斷」。
- 我不知道哪些欄位是必填、哪些欄位可以先留空。

### 我最怕的錯誤

- 我只是在轉述，系統卻把我當成親眼確認的人。
- 我的不確定描述被改寫成確定任務。

### 我不敢直接相信的資訊

- 模糊的地點、數量與需求類型。
- 來自社群或口述，但沒有明確來源證據的內容。

### 我希望系統提醒我的事

- 明確顯示「來源是口述 / 社群轉錄」與「目前未查核」。
- 在我填寫不確定資訊時，提醒我這些內容不能直接變成任務。

### 我想反問開發者的一個問題

- 如果我只知道一半內容，系統會不會逼我把整筆訊息都填成看起來很完整？

## Persona 2：資訊整理者

### 我想完成的事

- 把混亂資訊整理成下一位協作者能理解的內容。
- 保留哪些地方還不能判斷，而不是假裝已經確認。

### 我看不懂的地方

- 目前還無法一眼看出「原文」與「推論」之間的關係。
- 工作台裡的判斷結果和原始資料沒有足夠清楚的對照。

### 我最怕的錯誤

- 模糊資料被直接變成任務。
- AI 或整理者把原文沒有的資訊補進來，讓下一位協作者誤判。

### 我不敢直接相信的資訊

- 不明確的需求類型。
- 來源不清、時間不明、地點模糊的內容。

### 我希望系統提醒我的事

- 明確區分「原始內容」「判斷說明」「待人工確認」三者。
- 讓使用者知道哪些資訊是可疑、哪些還需要追問。

### 我想反問開發者的一個問題

- 如果一筆訊息目前沒有足夠證據，系統是否應該直接把它留在「待確認」而不是往下推？

## Persona 3：行動者

### 我想完成的事

- 知道哪些資訊可以直接採取下一步，哪些還不適合行動。
- 避免因為錯誤資訊而跑錯地方或做錯事。

### 我看不懂的地方

- 現在的畫面沒有很明確地把「可行動」「待確認」「不可直接採取行動」分開。
- 我不確定哪些內容是已確認、哪些只是候選判斷。

### 我最怕的錯誤

- 把未確認內容當成已確認。
- 到現場才發現任務已經不存在或方向錯誤。

### 我不敢直接相信的資訊

- 含糊的地址與時間。
- 可能已過期，卻沒有標示更新狀態的內容。

### 我希望系統提醒我的事

- 明確標示資訊是否足夠支持出任務或前往現場。
- 讓我一眼看出這筆資訊現在是否適合採取行動。

### 我想反問開發者的一個問題

- 如果一筆資訊是「可能」，系統有沒有辦法讓我不會把它誤解成「確定」？

## 人類補充

- 三個 sub-agent 模擬顯示，使用者最在意的其實不是「功能多不多」，而是「我能不能分辨什麼能信、什麼還不能信」。
- 這個原型已經把待人工確認與未查核顯示出來，但還需要更清楚的視覺層級，讓不同角色不必自己猜測。
- 我認為 v1 應該優先幫助「資訊整理者」與「行動者」理解狀態，而不是讓回報者一開始就被要求填完整資料。

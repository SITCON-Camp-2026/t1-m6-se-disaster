# Main Agent 訪談彙整

> 這份文件由 release kit 的 main-agent prompt 轉寫而成，保留了 persona 回饋的使用者視角，並加入人類判斷與取捨。

## 三個 persona 的共同需求

- 三個角色都希望快速分辨「這筆資訊現在能不能相信」。
- 他們都需要看見資訊來源與查核狀態，而不是只看到一段整理後文字。
- 他們都不希望不確定內容被伪裝成已確認事實。

## 三個 persona 之間的衝突需求

- 回報者想要越快越好，甚至希望能先把模糊資訊留下來。
- 資訊整理者需要更完整的上下文，才能判斷哪些資訊應該保留、哪些應該留給人工確認。
- 行動者則更關心「現在可不可以採取下一步」，因此希望畫面能更直接地顯示可行動性。

## Phase 0 原型中被推翻的假設

- 原本假設：使用者只要看得懂卡片與狀態標籤，就能自行判斷是否可行動。
- 被回饋推翻：實際上使用者非常依賴「來源說明」「查核狀態」「原文與判斷的區隔」，否則會把未確認資料誤解為已確認。

## 最容易造成誤導的風險

- 來源標籤與查核狀態被混淆，讓資料看起來比實際更可靠。
- 未確認內容被誤解成可執行任務。
- 使用者不知道哪些內容是原始訊息、哪些是整理者的推論。

## 可以考慮的設計方向

- 可以把「原始內容」「判斷說明」「下一步建議」拆成不同區塊，讓不同角色都能快速辨識。
- 可以在畫面上加上更強的視覺分層，讓「待確認」與「可行動」不至於混在一起。
- 可以針對不同角色給出不同的簡化視圖，但仍保留共同的安全邊界標記。

## 需要人類決策的取捨

- 取捨 1：要不要讓 v1 直接提供三種角色的專屬視圖。
  - 選項 A：只做一個共同視圖，所有角色共用同一套狀態與標籤。
  - 選項 B：為不同角色建立不同的簡化入口，但保留共同的原始資料與不確定標記。
  - 我目前傾向：選項 B。
  - 理由：不同角色對「可信任」與「可操作」的理解不同，單一視圖很容易讓某些人錯判。

## 不應該讓 AI 自動決定的事情

- 不應該把未確認內容自動改寫成已確認任務。
- 不應該在沒有來源證據時，替使用者補出地點、數量或需求類型。
- 不應該把「可能」與「已確認」混在同一個狀態內。

## 我採用的 AI 整理

- 我採用 AI 幫我把三個 persona 的回饋整理成固定格式，讓我能快速比較彼此差異。

## 我不採用的 AI 建議

- 建議：把所有訊息直接整理成任務清單，讓畫面看起來更像正式系統。
  - 不採用原因：這樣會把「待確認」與「可行動」混在一起，反而會放大誤判風險。

## 我自己補充的判斷

- v1 的核心價值，不是讓畫面看起來更完整，而是讓使用者能更清楚地分辨哪一段是原文、哪一段是判斷、哪一段還不該被當成事實。
- 這次訪談的重點不是新增更多功能，而是讓現有的資訊與狀態更清楚。

## 我仍不確定的地方

- 目前還不確定 v1 會先優先做「角色導向視圖」還是「狀態導向視圖」。
- 我們也還需要再確認，哪些欄位的缺失最值得在介面中直接被標註出來。

## 小隊討論後的補充

- 這次訪談把「不同角色」的需求拉出來之後，會比單純看一張卡片更容易知道 v1 該往哪裡聚焦。
